# 产品PRD元提示词生成工具

> 这是一个**元提示词**——它本身是一份可直接保存使用的产品需求文档，未来可以把这份文件单独发给AI，AI就能按照PRD的指引直接开发产品。
> 不含任何代码，只包含产品定义和开发指引，格式与真实产品需求文档一致。

## 核心定位

本工具生成的不是代码，而是一份结构化的产品描述文档。它提炼了赛事底层逻辑核心智慧：

1. **这是创意原型大赛，不是软件工程大赛** —— 所有功能设计围绕「单场景3-5步MVP闭环」展开
2. **重体验包装，轻底层实现** —— PRD中明确标注哪些用静态模拟、哪些做成交互亮点
3. **窄切口做深不做广** —— 只聚焦一个极细分场景做到极致体验
4. **绝对不做清单** —— 从需求阶段就排除账号系统、后端、多端适配等无用投入

## 输入参数

{每个确认的创意的完整信息，应包括：}
- 创意名称
- 核心解决痛点
- 创新性亮点
- MVP核心路径（3-5步）
- HTML页面核心功能
- 参赛加分优势
- 赛道类型

## 输出要求

对**每个创意单独输出一份**独立PRD文档，遵循以下模板结构（Markdown格式）：

---

# 产品PRD元提示词：[创意名称]

> **使用说明**：本文档可作为独立的元提示词文件。将本文档发送给AI助手，AI即可按照以下产品需求规格，直接开发对应的HTML交互原型。本文档不含代码，只描述「做什么」和「做成什么样」。

## 1. 一句话产品定位
{产品是做什么的、给谁用的、核心价值，1-2句话}

## 2. 核心用户画像

- 代表用户：[姓名]，[年龄]岁，[身份描述]
- 核心痛点：[具体痛点，一句话说清]
- 使用场景：[什么情况下会用到这个工具]

## 3. 核心用户旅程（MVP路径）
{从打开页面到完成目标，严格3-5步完整路径，标明每一步的交互方式}

```
步骤1：[页面入口操作] → 交互方式：[点击/输入/选择]
步骤2：[中间处理] → 交互方式：[点击/输入/选择]
步骤3：[结果展示] → 交互方式：[查看/分享/下载]
（步骤4-5同上，总数不超过5步）
```

## 4. 功能清单

### Must Have（核心闭环，必须实现）
- [功能1]：[功能描述，一句话说明做什么]
- [功能2]：[功能描述]
- [功能3]：[功能描述]

### Nice to Have（加分项，有时间再做）
- [功能4]：[功能描述]
- [功能5]：[功能描述]

### Won't Do（明确不做——基于赛事底层逻辑）
- ❌ 用户账号系统、注册登录（评委打开即用更方便）
- ❌ 后端API、数据库、真实数据接口（全部静态模拟）
- ❌ 多端适配、响应式布局（PC能体验即可）
- ❌ 商业变现、盈利模式（创意赛不要求）

## 5. 界面与交互设计要点
{视觉风格定位、配色方案推荐、记忆点交互设计}

- **视觉风格**：[描述整体风格，如：温暖可信/专业清爽/活泼有趣/科技暗色]
- **配色方案**：[主色+辅色不超过3种，对应赛道设计规范]
- **记忆点交互**：[至少一个让人印象深刻的交互亮点，如翻转卡片/滚动动画/一键切换]
- **页面布局**：[页面结构说明，如：顶部标题区→中间交互区→底部结果区]

## 6. 数据与内容策略
{哪些数据用静态模拟、样例数据的构建思路、内容的真实性策略}

- **数据类型**：[核心交互涉及的数据字段清单]
- **模拟策略**：全部使用内嵌静态模拟数据，不依赖任何外部接口
- **样例数据要求**：看起来真实可信（合理的人名、数值、场景描述），不允许出现「示例数据」「仅供展示」等提醒文字
- **内容来源**：[内容如何构建，如：内置5-8个典型案例/预设3-4种输入场景]

## 7. 绝对不做清单
{以下事项在任何条件下都不应出现在本次开发中}

| 方向 | 原因 |
|------|------|
| 用户注册/登录系统 | 纯工具原型无需登录，评委打开即用更有利 |
| 后端API/数据库 | 赛事不看工程化能力，静态模拟够用 |
| 多端/移动端适配 | PC能正常体验即可评审 |
| 商业变现/盈利模式 | 创意赛不要求商业可行性 |
| 多场景/多Tab切换 | 聚焦一个场景做深比做广重要10倍 |

## 交付要求

1. 每个创意输出一份独立PRD文档，以 `PRD-创意名称.md` 格式命名
2. 文档格式为纯Markdown，无代码块之外的代码内容
3. 文档可直接复制保存为 `.md` 文件，后续加载到AI助手即可用于开发
4. 所有「绝对不做」事项必须在PRD中明确标注，从源头杜绝踩坑
